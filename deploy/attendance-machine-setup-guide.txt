  BƯỚC 1: KIỂM TRA BACKEND ĐÃ SẴN SÀNG NHẬN DỮ LIỆU
================================================================================

SSH vào VPS:
     ssh root@123.30.48.72
     cd /root/erp-system

1.1. Kiểm tra backend đang chạy:
     docker compose ps

     → Tất cả services (backend, nginx, db, redis) phải ở trạng thái "Up"

1.2. Kiểm tra endpoint iclock hoạt động qua HTTP:
     curl -v "http://api.hrmgo.site/iclock/cdata?SN=TEST123"

     → Response ĐÚNG phải trả về dạng text/plain, nội dung kiểu:
          GET OPTION FROM: TEST123
          Stamp=0
          OpStamp=0
          PhotoStamp=0
          ErrorDelay=30
          Delay=10
          TransTimes=00:00;14:05
          TransInterval=1
          TransFlag=TransData AttLog	OpLog	AttPhoto	EnrollUser	EnrollFP
          Realtime=1
          TimeZone=7
          ServerVer=2.4.1

     Giải thích các tham số server trả về:
     - Stamp=0: Máy gửi tất cả records (không lọc theo thời gian)
     - Realtime=1: Máy gửi ngay khi có chấm công mới (không batch)
     - ErrorDelay=30: Nếu lỗi, máy thử lại sau 30 giây
     - Delay=10: Máy polling lệnh mới mỗi 10 giây
     - TimeZone=7: Múi giờ GMT+7 (Việt Nam)

     → Nếu trả về HTML hoặc lỗi 400:
          - Kiểm tra ALLOWED_HOSTS trong backend/.env.production
            phải chứa api.hrmgo.site
          - Rebuild: docker compose up -d --build backend

     → Nếu connection refused:
          - Kiểm tra nginx đang chạy: docker compose ps
          - Kiểm tra port 80 mở: ufw status

1.3. Kiểm tra endpoint getrequest:
     curl -v "http://api.hrmgo.site/iclock/getrequest?SN=TEST123"

     → Response phải là: OK
     (Nghĩa là server không có lệnh nào cần gửi cho máy)

1.4. Kiểm tra log backend:
     docker compose logs backend --tail 20

     → Phải thấy dòng log kiểu:
          [iclock] Handshake từ SN=TEST123
     (Chứng tỏ request đã đi qua nginx vào tới Django)

  XỬ LÝ SỰ CỐ MÁY CHẤM CÔNG
================================================================================

─── SỰ CỐ 0: Web hiện "Thiết bị đang kết nối" nhưng SN=healthcheck ───

Nguyên nhân:
  Docker healthcheck gọi /iclock/cdata?SN=healthcheck mỗi 30 giây
  từ 127.0.0.1. Hệ thống nhầm tưởng là máy chấm công thật.

  Triệu chứng:
  - Trang Device Settings hiện: Số Serial = healthcheck
  - IP thiết bị = 127.0.0.1
  - Push cuối cập nhật liên tục mỗi 30s
  - Nhưng KHÔNG có dữ liệu chấm công thật

  → ĐÂY KHÔNG PHẢI MÁY CHẤM CÔNG THẬT, chỉ là Docker healthcheck.

  Đã sửa code (zk_service.py): Bỏ qua SN=healthcheck trong device registry.
  Sau khi deploy code mới, trang web sẽ hiện đúng trạng thái:
  "Đang chờ máy chấm công kết nối" cho đến khi máy CC thật gửi request.

  Để deploy fix:
  1. Push code lên GitHub
  2. Trên VPS: cd /root/erp-system && git pull
  3. docker compose up -d --build backend
  4. Refresh trang Device Settings → phải hiện "Đang chờ kết nối"

─── SỰ CỐ A: Máy CC không kết nối, không thấy log handshake ───

Nguyên nhân có thể:
  1. Máy CC không có internet
  2. DNS không resolve được api.hrmgo.site
  3. Port 80 bị chặn
  4. Cấu hình Cloud Server sai

Xử lý theo thứ tự:

  a) Kiểm tra mạng máy CC:
     - Máy CC có đèn mạng sáng không?
     - Thử ping từ máy tính cùng mạng LAN:
       ping 192.168.1.225  (IP máy CC)
     - Nếu không ping được → kiểm tra cáp mạng/WiFi

  b) Kiểm tra máy CC resolve được domain:
     - Từ máy tính cùng mạng LAN:
       nslookup api.hrmgo.site
     - Nếu không resolve → DNS trên máy CC hoặc router chưa đúng
     - Thử cấu hình DNS trên máy CC: 8.8.8.8

  c) Kiểm tra port 80 trên VPS:
     (Trên VPS)
     ufw status
     → Port 80 phải ALLOW từ Anywhere

     Nếu chưa:
     ufw allow 80/tcp
     ufw reload

  d) Kiểm tra nginx proxy /iclock/ qua HTTP:
     curl -v "http://api.hrmgo.site/iclock/cdata?SN=TEST"
     → Phải trả về text/plain với Stamp=0, Realtime=1,...

  e) Nếu curl từ VPS OK nhưng máy CC không kết nối:
     → Vấn đề nằm ở mạng giữa máy CC và internet
     → Kiểm tra router/firewall của mạng LAN

─── SỰ CỐ B: Handshake OK nhưng không nhận được chấm công (ATTLOG) ───

Nguyên nhân:
  1. Máy CC chưa có dữ liệu chấm công mới

Xử lý:
  a) Chấm công thử trực tiếp trên máy CC
  b) Đợi 10-30 giây (Delay=10 trong handshake config)
  c) Kiểm tra log: docker compose logs backend -f --tail 30
  d) Nếu vẫn không thấy ATTLOG:
     - Khởi động lại máy CC
     - Máy sẽ handshake lại và push toàn bộ record chưa gửi

─── SỰ CỐ C: Log nhận ATTLOG nhưng web không hiển thị ───

Nguyên nhân:

     - Kiểm tra nginx config có block /ws/

─── SỰ CỐ E: Dữ liệu chấm công bị trùng lặp ───

Nguyên nhân: Máy CC gửi lại record đã tồn tại (ví dụ sau khi khởi động lại)

  - Không cần xử lý gì thêm

─── SỰ CỐ F: Muốn dùng IP thay vì domain cho máy CC ───

Nếu máy CC firmware cũ không hỗ trợ domain name:
  1. Cấu hình trên máy CC: Server = 123.30.48.72, Port = 80
  2. Trên VPS, thêm IP vào ALLOWED_HOSTS:

================================================================================
  GIAO THỨC ICLOCK (ADMS) - CHI TIẾT KỸ THUẬT
================================================================================

  Máy chấm công Ronald Jack AI06F giao tiếp với server qua HTTP,
  sử dụng giao thức iclock (hay ADMS push mode).

  LUỒNG GIAO TIẾP:
  ─────────────────
  1. Máy bật → GET /iclock/cdata?SN=<serial> (Handshake)
     ← Server trả cấu hình: Stamp, TransFlag, Realtime...

  2. Máy push data → POST /iclock/cdata?SN=<serial>&table=ATTLOG
     Body: dữ liệu tab-separated
     ← Server trả: OK: <số record đã lưu>

  3. Máy polling lệnh → GET /iclock/getrequest?SN=<serial>
     ← Server trả: OK (không có lệnh) hoặc lệnh cụ thể

  4. Máy xác nhận lệnh → POST /iclock/devicecmd?SN=<serial>
     Body: kết quả thực thi lệnh

  ENDPOINTS:
  ──────────
  ┌─────────────────────────────────────────────┬──────────┬──────────────────────┐
  │ URL                                         │ Method   │ Mục đích             │
  ├─────────────────────────────────────────────┼──────────┼──────────────────────┤
  │ /iclock/cdata?SN=xxx                        │ GET      │ Handshake            │
  │ /iclock/cdata?SN=xxx&table=ATTLOG           │ POST     │ Push chấm công       │
  │ /iclock/cdata?SN=xxx&table=OPERLOG          │ POST     │ Push log vận hành    │
  │ /iclock/cdata?SN=xxx&table=USERINFO         │ POST     │ Push thông tin NV    │
  │ /iclock/getrequest?SN=xxx                   │ GET      │ Máy xin lệnh        │
  │ /iclock/devicecmd?SN=xxx                    │ POST     │ Xác nhận lệnh       │
  │ /cdata?SN=xxx                               │ GET/POST │ Alias (tương thích)  │
  └─────────────────────────────────────────────┴──────────┴──────────────────────┘

  ĐỊNH DẠNG ATTLOG (dữ liệu chấm công):
  ───────────────────────────────────────
  Mỗi dòng = 1 lần chấm công, các trường phân cách bằng TAB:

  UserID<TAB>DateTime<TAB>Verified<TAB>Status<TAB>WorkCode

  Ví dụ:
  1	2026-04-03 08:05:33	1	0	0
  2	2026-04-03 08:10:12	1	1	0

  ┌──────────┬──────────────────────────────────────────────┐
  │ Trường   │ Ý nghĩa                                      │
  ├──────────┼──────────────────────────────────────────────┤
  │ UserID   │ Mã NV trên máy CC (khớp với Employee.user_id)│
  │ DateTime │ YYYY-MM-DD HH:MM:SS hoặc YYYY/MM/DD HH:MM:SS│
  │ Verified │ 0=chưa xác thực, 1=đã xác thực              │
  │ Status   │ 0=check-in, 1=check-out, 2=break,           │
  │          │ 3=return, 4-5=overtime                        │
  │ WorkCode │ Mã công (thường = 0)                         │
  └──────────┴──────────────────────────────────────────────┘
