  BƯỚC 1: KIỂM TRA BACKEND ĐÃ SẴN SÀNG NHẬN DỮ LIỆU
================================================================================

SSH vào VPS:
     ssh root@123.30.48.72
     cd /opt/erp

1.1. Kiểm tra backend đang chạy:
     docker compose ps

     → Tất cả services (backend, nginx, db, redis) phải ở trạng thái "Up"

1.2. Kiểm tra endpoint iclock hoạt động qua HTTP:
     curl -v "http://api.hrmgo.site/iclock/cdata?SN=TEST123"

     → Response ĐÚNG phải trả về dạng text/plain, nội dung kiểu:
          GET OPTION FROM: TEST123
          Stamp=0
          OpStamp=0
          PhotoStamp=0
          ErrorDelay=30
          Delay=10
          TransTimes=00:00;14:05
          TransInterval=1
          TransFlag=TransData AttLog	OpLog	AttPhoto	EnrollUser	EnrollFP
          Realtime=1
          TimeZone=7
          ServerVer=2.4.1

     Giải thích các tham số server trả về:
     - Stamp=0: Máy gửi tất cả records (không lọc theo thời gian)
     - Realtime=1: Máy gửi ngay khi có chấm công mới (không batch)
     - ErrorDelay=30: Nếu lỗi, máy thử lại sau 30 giây
     - Delay=10: Máy polling lệnh mới mỗi 10 giây
     - TimeZone=7: Múi giờ GMT+7 (Việt Nam)

     → Nếu trả về HTML hoặc lỗi 400:
          - Kiểm tra ALLOWED_HOSTS trong backend/.env.production
            phải chứa api.hrmgo.site
          - Rebuild: docker compose up -d --build backend

     → Nếu connection refused:
          - Kiểm tra nginx đang chạy: docker compose ps
          - Kiểm tra port 80 mở: ufw status

1.3. Kiểm tra endpoint getrequest:
     curl -v "http://api.hrmgo.site/iclock/getrequest?SN=TEST123"

     → Response phải là: OK
     (Nghĩa là server không có lệnh nào cần gửi cho máy)

1.4. Kiểm tra log backend:
     docker compose logs backend --tail 20

     → Phải thấy dòng log kiểu:
          [iclock] Handshake từ SN=TEST123
     (Chứng tỏ request đã đi qua nginx vào tới Django)

  BƯỚC 2: KIỂM TRA CẤU HÌNH .env.production TRÊN VPS
================================================================================

2.1. Kiểm tra biến môi trường liên quan máy chấm công:
     grep "ZK_" backend/.env.production

     Nếu chưa có, thêm vào file:
     nano backend/.env.production

     Thêm các dòng sau (cuối file):
     ──────────────────────────────
     ZK_DEVICE_IP=192.168.1.225
     ZK_DEVICE_PORT=5005
     ZK_DEVICE_TIMEOUT=10
     ZK_DEVICE_PASSWORD=0
     ZK_PROTOCOL=adms
     ──────────────────────────────

     Giải thích:
     - ZK_DEVICE_IP: IP máy chấm công trên mạng LAN
       (dùng cho chế độ pull trực tiếp, không cần thiết cho ADMS push)
     - ZK_DEVICE_PORT: 5005 cho Ronald Jack ADMS
     - ZK_PROTOCOL=adms: Chế độ ADMS push (máy tự gửi data lên server)

     LƯU Ý: Với chế độ ADMS push qua internet (máy CC → api.hrmgo.site),
     các biến ZK_DEVICE_IP, ZK_DEVICE_PORT chỉ mang tính tham chiếu.
     Máy CC sẽ tự kết nối tới server thông qua cấu hình trên máy (bước 3).

     Sau khi thêm, lưu file (Ctrl+O, Enter, Ctrl+X) rồi rebuild:
     docker compose up -d --build backend

     ├─────────────────────┼──────────────────┼──────────────────────────┤
     │ Yêu cầu máy ch      │ Có               │ Có  (giữ nguyên)        │
     │ Sử dụng tên miền    │ Không            │ Có  ← ĐỔI               │
     │ Tên miền            │ www.yunatt.com    │ api.hrmgo.site  ← ĐỔI   │

     │ Ủy quyền máy c      │ Không            │ Không (giữ nguyên)      │
     └─────────────────────┴──────────────────┴──────────────────────────┘

     CHỈ CẦN ĐỔI 3 TRƯỜNG:
     ═══════════════════════

     2. "Tên miền"           →  xóa www.yunatt.com, nhập: api.hrmgo.site

     3. "Số cổng Server"     →  đổi từ 8000 → 80
        (Port 80 = HTTP, nginx trên VPS listen port 80 cho /iclock/)

     CÁC TRƯỜNG KHÁC GIỮ NGUYÊN:
     - "Yêu cầu máy ch" = Có (đã bật, giữ nguyên)
     - "IP server" = không quan trọng khi đã bật "Sử dụng tên miền"
     - "Tốc độ" = 3 (giữ nguyên)
     - "Ủy quyền máy c" = Không (không dùng proxy)

     GIẢI THÍCH:
     - Port 80: Nginx trên VPS listen port 80 cho path /iclock/
       và proxy vào Django backend (port 8000 nội bộ container)
     - Port 8000 chỉ dùng khi kết nối trực tiếp Django trên LAN
     - Máy CC ADMS chỉ hỗ trợ HTTP, KHÔNG hỗ trợ HTTPS (443)

     - Status: 0=check-in, 1=check-out, 2=break, 3=return

================================================================================
  GIAO THỨC ICLOCK (ADMS) - CHI TIẾT KỸ THUẬT
================================================================================

  Máy chấm công Ronald Jack AI06F giao tiếp với server qua HTTP,
  sử dụng giao thức iclock (hay ADMS push mode).

  LUỒNG GIAO TIẾP:
  ─────────────────
  1. Máy bật → GET /iclock/cdata?SN=<serial> (Handshake)
     ← Server trả cấu hình: Stamp, TransFlag, Realtime...

  2. Máy push data → POST /iclock/cdata?SN=<serial>&table=ATTLOG
     Body: dữ liệu tab-separated
     ← Server trả: OK: <số record đã lưu>

  3. Máy polling lệnh → GET /iclock/getrequest?SN=<serial>
     ← Server trả: OK (không có lệnh) hoặc lệnh cụ thể

  4. Máy xác nhận lệnh → POST /iclock/devicecmd?SN=<serial>
     Body: kết quả thực thi lệnh

  ENDPOINTS:
  ──────────
  ┌─────────────────────────────────────────────┬──────────┬──────────────────────┐
  │ URL                                         │ Method   │ Mục đích             │
  ├─────────────────────────────────────────────┼──────────┼──────────────────────┤
  │ /iclock/cdata?SN=xxx                        │ GET      │ Handshake            │
  │ /iclock/cdata?SN=xxx&table=ATTLOG           │ POST     │ Push chấm công       │
  │ /iclock/cdata?SN=xxx&table=OPERLOG          │ POST     │ Push log vận hành    │
  │ /iclock/cdata?SN=xxx&table=USERINFO         │ POST     │ Push thông tin NV    │
  │ /iclock/getrequest?SN=xxx                   │ GET      │ Máy xin lệnh        │
  │ /iclock/devicecmd?SN=xxx                    │ POST     │ Xác nhận lệnh       │
  │ /cdata?SN=xxx                               │ GET/POST │ Alias (tương thích)  │
  └─────────────────────────────────────────────┴──────────┴──────────────────────┘

  ĐỊNH DẠNG ATTLOG (dữ liệu chấm công):
  ───────────────────────────────────────
  Mỗi dòng = 1 lần chấm công, các trường phân cách bằng TAB:

  UserID<TAB>DateTime<TAB>Verified<TAB>Status<TAB>WorkCode

  Ví dụ:
  1	2026-04-03 08:05:33	1	0	0
  2	2026-04-03 08:10:12	1	1	0

  ┌──────────┬──────────────────────────────────────────────┐
  │ Trường   │ Ý nghĩa                                      │
  ├──────────┼──────────────────────────────────────────────┤
  │ UserID   │ Mã NV trên máy CC (khớp với Employee.user_id)│
  │ DateTime │ YYYY-MM-DD HH:MM:SS hoặc YYYY/MM/DD HH:MM:SS│
  │ Verified │ 0=chưa xác thực, 1=đã xác thực              │
  │ Status   │ 0=check-in, 1=check-out, 2=break,           │
  │          │ 3=return, 4-5=overtime                        │
  │ WorkCode │ Mã công (thường = 0)                         │
  └──────────┴──────────────────────────────────────────────┘

SECRET_KEY DÙNG Ở ĐÂU TRONG CODE?
────────────────────────────────────

  SECRET_KEY được Django dùng để:
  - Ký JWT tokens / session cookies (đăng nhập)
  - Tạo CSRF tokens (bảo vệ form submit)
  - Các tác vụ mã hóa nội bộ

  TRONG CODEBASE NÀY, SECRET_KEY xuất hiện ở 2 nơi:

  1. backend/config/settings_prod.py  (production — ĐANG DÙNG trên VPS)
     → Dòng 13: SECRET_KEY = os.environ['SECRET_KEY']
     → Đọc từ biến môi trường, KHÔNG hardcode
     → ĐÂY là file chạy thật trên VPS

  2. backend/config/settings.py  (development — chỉ dùng khi localhost)
     → Dòng 31: SECRET_KEY = 'django-insecure-d)sjr@v_...'
     → Key hardcode cho môi trường dev, KHÔNG ảnh hưởng production
     → KHÔNG cần sửa file này

  KẾT LUẬN:
  Chỉ cần cập nhật 1 chỗ duy nhất:
  → File backend/.env.production trên VPS  (dòng SECRET_KEY=...)
  → Sau đó rebuild backend → Django sẽ đọc key mới


LÀM SAO BIẾT SECRET_KEY ĐANG BỊ HỎNG?
────────────────────────────────────────

  Triệu chứng SECRET_KEY bị lỗi:
  - Đăng nhập xong bị logout ngay lập tức
  - Không giữ được session đăng nhập
  - Lỗi 500 Internal Server Error khi submit form
  - Token JWT không hợp lệ

  Cách kiểm tra nhanh xem SECRET_KEY hiện tại có bị bẩn không:
  (SSH vào VPS)
    grep "SECRET_KEY" backend/.env.production

  → Nếu thấy dòng có chứa chữ "root@" hoặc ":/opt/erp#" hoặc bất kỳ
    ký tự shell nào → KEY BỊ HỎNG, cần tạo lại ngay.

  Ví dụ key BỊ HỎNG (có shell prompt dính vào):
    SECRET_KEY=xUlRY0KuTO3Hc1ExUDitOxMRdbPJsYeRTqHPCwQJhbEQkmNuFKroot@control-erp-hrm:/opt/erp#

  Ví dụ key ĐÚNG (chỉ có ký tự chữ, số, ký hiệu):
    SECRET_KEY=aB3xK9mP2nQr7sT4vW1yZ6cD8eF5gH0jIlLo


CÁC BƯỚC TẠO SECRET_KEY MỚI VÀ CẬP NHẬT
──────────────────────────────────────────

  ═══ BƯỚC 1: SSH vào VPS ═══

    ssh root@123.30.48.72
    cd /opt/erp

  ═══ BƯỚC 2: Tạo SECRET_KEY mới (chạy lệnh này) ═══

    python3 -c 'import secrets; print(secrets.token_urlsafe(50))'

    LƯU Ý: Dùng dấu NHÁY ĐƠN '...' bao quanh lệnh python, KHÔNG dùng nháy kép.
    (Nháy kép khiến bash xử lý ký tự ! gây lỗi "event not found")

    → Lệnh này tạo ra 1 chuỗi ngẫu nhiên ~67 ký tự (A-Z, a-z, 0-9, -, _), ví dụ:
      aB3xK9mP2nQr7sT4vW1yZ6cD8eF5gH0jIlLoMpNqRsUuVwXyZa1b2c3d4e5f6g7h

    QUAN TRỌNG:
    - Copy toàn bộ chuỗi đó (chỉ copy chuỗi, KHÔNG copy dấu nhắc lệnh)
    - KHÔNG chia sẻ chuỗi này với bất kỳ ai
    - Lưu ở chỗ an toàn (ví dụ file notepad riêng)

  ═══ BƯỚC 3: Mở file .env.production để chỉnh sửa ═══

    nano backend/.env.production

  ═══ BƯỚC 4: Tìm dòng SECRET_KEY và sửa ═══

    Tìm dòng bắt đầu bằng SECRET_KEY=...
    (Dùng Ctrl+W trong nano để tìm nhanh: Ctrl+W → gõ SECRET_KEY → Enter)

    Xóa toàn bộ nội dung dòng đó (từ SECRET_KEY= đến hết dòng)
    Nhập lại:
      SECRET_KEY=<CHUỖI_VỪA_TẠO_Ở_BƯỚC_2>

    Ví dụ:
      SECRET_KEY=aB3xK9mP2nQr7sT4vW1yZ6cD8eF5gH0jIlLoMpNqRsUuVwXyZ12345678

    LƯU Ý BẮT BUỘC:
    - KHÔNG có dấu cách trước hoặc sau dấu =
    - KHÔNG có dấu nháy đơn hay nháy kép bao quanh
    - Dòng phải kết thúc ngay sau ký tự cuối cùng của key
    - KHÔNG được có text thừa như "root@..." hay ":/opt/erp#"

  ═══ BƯỚC 5: Lưu file ═══

    Ctrl+O → Enter → Ctrl+X

  ═══ BƯỚC 6: Kiểm tra lại trước khi rebuild ═══

    grep "SECRET_KEY" backend/.env.production

    → Phải thấy đúng 1 dòng, dạng:
      SECRET_KEY=<60_ký_tự_sạch>

    → KHÔNG có root@, KHÔNG có :/, KHÔNG có khoảng trắng thừa

  ═══ BƯỚC 7: Rebuild backend ═══

    docker compose up -d --build backend

    Đợi 30-60 giây cho backend khởi động lại.

  ═══ BƯỚC 8: Kiểm tra backend đã khởi động thành công ═══

    docker compose logs backend --tail 20

    → Phải thấy dòng: "Starting Daphne..."
    → KHÔNG có dòng lỗi KeyError hoặc ImproperlyConfigured

    docker compose ps

    → backend phải ở trạng thái "Up (healthy)"

  ═══ BƯỚC 9: Test đăng nhập ═══

    Mở trình duyệt → https://hrmgo.site
    Đăng nhập với tài khoản admin
    → Phải vào được và giữ được phiên đăng nhập

    Nếu login lỗi 500:
    docker compose logs backend --tail 50
    → Xem dòng lỗi cuối cùng để debug

    - run_zk_pull (port 4370): pull định kỳ từ máy CC (backup đáng tin cậy)

  Sơ đồ kết nối:

    VPS (123.30.48.72)
          │
          │  ZKBinaryService._connect()
          │  "113.160.150.125:4370"
          │
          ▼
    Router MikroTik (WAN: 113.160.150.125)
          │
          │  NAT dstnat: dst-port=4370 → to-addresses=192.168.1.225:4370
          │
          ▼
    Máy CC Ronald Jack AI06F (LAN: 192.168.1.225:4370)
          │
          │  pyzk binary protocol
          │  conn.get_attendance() → tất cả bản ghi chấm công
          │
          ▼
    Django lưu vào DB (get_or_create → không tạo trùng)
    → WebSocket push realtime lên frontend

    ✅ Timestamp CHÍNH XÁC từ máy (không do server tạo)
    ✅ Lấy được TOÀN BỘ log trên máy CC
    ✅ Không bị mất data khi notification cache bị kẹt
    ✅ Idempotent: pull bao nhiêu lần cũng không tạo duplicate

  Điều kiện đã xác nhận:
    ✅ IP WAN 113.160.150.125 — IP TĨNH đã đăng ký với VNPT

  BƯỚC 1 — MIKROTIK: THÊM NAT PORT FORWARDING
  ─────────────────────────────────────────────

  Mở WinBox → đăng nhập router 192.168.1.1
  Vào: New Terminal (menu trái)

  Chạy lệnh sau trong terminal MikroTik:

    /ip firewall nat add \
      chain=dstnat \
      dst-address=113.160.150.125 \
      protocol=tcp \
      dst-port=4370 \
      action=dst-nat \
      to-addresses=192.168.1.225 \
      to-ports=4370 \
      comment="ZK attendance machine port 4370"

  Xác nhận rule đã được tạo:

    /ip firewall nat print

  → Phải thấy rule mới xuất hiện với:
      dst-port=4370
      to-addresses=192.168.1.225
      to-ports=4370

  LƯU Ý VỀ RULE HIỆN CÓ (tham chiếu từ màn hình WinBox):
    #0 dstnat → 113.160.150.125 tcp:8088   (may chu hoadon)
    #1 dstnat → 113.160.150.125 tcp:8090   (tra cuu hoa don)
    #5 dstnat → 113.160.150.125 tcp:22     (SSH)
    → Rule mới (port 4370) sẽ thêm vào danh sách này

  KIỂM TRA PORT FORWARDING (từ bên ngoài mạng LAN):

    Từ VPS hoặc máy khác ngoài internet:
    nc -zv 113.160.150.125 4370

    → Nếu "Connected": port forwarding hoạt động đúng
    → Nếu "Connection refused" hoặc timeout: kiểm tra lại rule MikroTik

  XỬ LÝ SỰ CỐ
  ────────────

  ❌ "Connection refused" (port 4370):
     → MikroTik NAT rule chưa đúng, hoặc máy CC không lắng nghe port 4370
     → Kiểm tra: /ip firewall nat print trên MikroTik
     → Thử kết nối LAN: nc -zv 192.168.1.225 4370

  ❌ "Connection timed out":
     → Firewall VPS hoặc MikroTik chặn kết nối outbound
     → Kiểm tra ufw trên VPS: ufw status (port outbound không bị chặn)

  ❌ "ZK binary connect failed: No module named 'zk'":
     → pyzk chưa được cài. Chạy: docker compose up -d --build backend

  ❌ Pull thành công nhưng 0 bản ghi mới:
     → Tất cả dữ liệu đã có trong DB rồi (bình thường khi pull lần 2+)
     → Kiểm tra: SELECT COUNT(*) FROM attendance_attendancelog;

  ❌ Timestamp sai giờ (lệch 7h):
     → Trong zk_service.py, hàm get_all_attendance() trả timestamp dạng string
        không có timezone info. Kiểm tra xem Django đang dùng timezone.make_aware()
        với đúng timezone (Asia/Ho_Chi_Minh) chưa.

  ❌ ZK Pull chạy nhưng ZK Push bị crash:
     → 2 tiến trình hoàn toàn độc lập, một cái crash không ảnh hưởng cái kia
     → Kiểm tra log riêng: docker compose logs backend | grep "ZK-PUSH\|ZK-PULL"
