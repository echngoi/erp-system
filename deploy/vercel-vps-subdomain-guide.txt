  Kiến trúc:
    ┌─────────────────┐         ┌──────────────────────────────┐
    │  hrmgo.site      │         │  api.hrmgo.site              │
    │  (Vercel)        │  ────►  │  (VPS 123.30.48.72)          │
    │  React Frontend  │  CORS   │  Django API + WebSocket      │

  Domain chính:
    - hrmgo.site, www.hrmgo.site  →  Vercel (React Frontend)

================================================================================
  BƯỚC 1: CẤU HÌNH DNS TẠI NHÀ CUNG CẤP DOMAIN
================================================================================

Vào trang quản lý DNS của domain hrmgo.site (nơi bạn mua domain).

1.1. XÓA các bản ghi A cũ nếu có:
     - Xóa bản ghi A của @ (root) trỏ về 123.30.48.72 (VPS)
     - Xóa bản ghi A của www trỏ về 123.30.48.72

1.2. THÊM bản ghi cho VERCEL (Frontend):
     ┌──────────┬──────────┬─────────────────────────┐
     │ Type     │ Host     │ Value                   │
     ├──────────┼──────────┼─────────────────────────┤
     │ A        │ @        │ 216.198.79.1            │
     │ CNAME    │ www      │ cname.vercel-dns.com.   │
     └──────────┴──────────┴─────────────────────────┘

     Giải thích:
     - Với project/domain của bạn, hãy ưu tiên đúng giá trị A record mà Vercel Dashboard đang hiển thị
     - Hiện tại dashboard của bạn đang yêu cầu 216.198.79.1 thì dùng 216.198.79.1
     - Không nên cố chấp dùng một IP "chung" lấy từ tài liệu cũ nếu dashboard đang khuyến nghị IP khác
     - CNAME www → cname.vercel-dns.com cho www.hrmgo.site

1.3. THÊM bản ghi cho VPS (Backend API):
     ┌──────────┬──────────┬─────────────────────────┐
     │ Type     │ Host     │ Value                   │
     ├──────────┼──────────┼─────────────────────────┤
     │ A        │ api      │ 123.30.48.72            │
     └──────────┴──────────┴─────────────────────────┘

     Kết quả:
     - api.hrmgo.site → 123.30.48.72 (VPS)

1.4. KIỂM TRA DNS đã propagate:
     (Chạy trên máy tính cá nhân, KHÔNG phải VPS)

     nslookup hrmgo.site
     → Phải trả về: 216.198.79.1

     nslookup api.hrmgo.site
     → Phải trả về: 123.30.48.72

     Nếu chưa đúng, đợi 5-30 phút rồi kiểm tra lại.
     Có thể dùng web: https://dnschecker.org/#A/api.hrmgo.site

2.2. Chọn project "erp-system" (hoặc tên project của bạn)

2.4. Thêm domain:

     - Vercel thường tự động đề xuất option này khi thêm www

================================================================================
  BƯỚC 4: CẬP NHẬT NGINX TRÊN VPS (CHO api.hrmgo.site)
================================================================================

SSH vào VPS:
     ssh root@123.30.48.72

4.1. Sửa file nginx.conf:
     cd /root/erp-system
     nano deploy/nginx.conf

     → Thay TOÀN BỘ nội dung file bằng nội dung dưới đây:
           → CHỈ copy phần cấu hình nginx thực tế
           → KHÔNG copy các dòng trang trí như:
                ─── BẮT ĐẦU NỘI DUNG deploy/nginx.conf ───
                ─── KẾT THÚC NỘI DUNG deploy/nginx.conf ───
           → Nếu copy cả các dòng này, nginx sẽ lỗi:
                unknown directive "───"

─── BẮT ĐẦU NỘI DUNG deploy/nginx.conf ───

# ───────────────────────────────────────────────────────────
# Nginx config for ERP Production - API Subdomain
# - api.hrmgo.site: Backend API + WebSocket + iclock
# - Frontend served by Vercel (hrmgo.site)
# ───────────────────────────────────────────────────────────

# ── Rate limiting cho iclock (chống spam) ──
limit_req_zone $binary_remote_addr zone=iclock:10m rate=30r/s;

# ── Upstream ──
upstream django {
    server backend:8000;
}

# ══════════════════════════════════════════════════════════
# HTTP (Port 80)
# - Máy chấm công ADMS chỉ hỗ trợ HTTP
# - ACME challenge cho Let's Encrypt
# - Mọi request khác redirect → HTTPS
# ══════════════════════════════════════════════════════════
server {
    listen 80;
    server_name api.hrmgo.site;

    # Let's Encrypt challenge
    location /.well-known/acme-challenge/ {
        root /var/www/certbot;
    }

    # ── iclock: Máy chấm công push data (HTTP only) ──
    location /iclock/ {
        limit_req zone=iclock burst=50 nodelay;
        proxy_pass http://django;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto http;
        proxy_connect_timeout 30s;
        proxy_read_timeout 60s;
    }

    location /cdata {
        limit_req zone=iclock burst=50 nodelay;
        proxy_pass http://django;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto http;
    }

    # ── Tất cả request khác → HTTPS ──
    location / {
        return 301 https://$host$request_uri;
    }
}

# ══════════════════════════════════════════════════════════
# HTTPS (Port 443)
# - API, WebSocket, Admin, Static, Media
# ══════════════════════════════════════════════════════════
server {
    listen 443 ssl http2;
    server_name api.hrmgo.site;

    # SSL certificates (Let's Encrypt)
    ssl_certificate /etc/letsencrypt/live/api.hrmgo.site/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.hrmgo.site/privkey.pem;

    # SSL Security
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # Security headers
    add_header X-Frame-Options "SAMEORIGIN" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Strict-Transport-Security "max-age=63072000" always;

    # Request size limit (cho file upload)
    client_max_body_size 10M;

    # ── API Django ──
    location /api/ {
        proxy_pass http://django;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }

    # ── Django Admin ──
    location /admin/ {
        proxy_pass http://django;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }

    # ── WebSocket ──
    location /ws/ {
        proxy_pass http://django;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto https;
        proxy_read_timeout 86400;
    }

    # ── iclock qua HTTPS (fallback nếu máy CC hỗ trợ) ──
    location /iclock/ {
        proxy_pass http://django;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }

    # ── Static files (Django collectstatic) ──
    location /static/ {
        alias /var/www/static/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # ── Media files ──
    location /media/ {
        alias /var/www/media/;
        expires 7d;
    }

    # ── Health check ──
    location /health {
        return 200 'OK';
        add_header Content-Type text/plain;
    }

    # ── Catch-all → 404 (không serve FE nữa) ──
    location / {
        return 404;
    }
}

─── KẾT THÚC NỘI DUNG deploy/nginx.conf ───

     → Phải thấy response JSON (405 hoặc 400 đều OK, nghĩa là Django đang nhận request)

                thì vẫn chấp nhận được ở bước này. Endpoint login thường cần POST + JSON body,

       Nếu localhost trả response nhưng domain không → DNS chưa propagate

5.4. Lấy SSL certificate:
           QUAN TRỌNG:
           Service certbot trong docker-compose.yml đang có entrypoint để chạy
           "certbot renew" tự động. Vì vậy nếu dùng:
           docker compose run --rm certbot certonly ...
           thì rất dễ bị chạy nhầm renewal thay vì certonly.

           Hãy dùng đúng lệnh này để bỏ qua entrypoint hiện tại:

           docker compose run --rm --entrypoint certbot certbot certonly \
       --webroot \
       --webroot-path=/var/www/certbot \
       -d api.hrmgo.site \
       --email your-email@gmail.com \
       --agree-tos \
       --no-eff-email

     → Thành công sẽ hiện: "Successfully received certificate"
     → Certificate lưu tại: /etc/letsencrypt/live/api.hrmgo.site/
           → Kiểm tra lại bằng:
                docker compose exec nginx ls /etc/letsencrypt/live/api.hrmgo.site/
                Nếu thấy fullchain.pem và privkey.pem thì cert đã tạo xong

     LƯU Ý: Thay your-email@gmail.com bằng email thật của bạn.

           Nếu bạn thấy output kiểu:
           "Processing /etc/letsencrypt/renewal/hrmgo.site.conf"
           "Certificate not yet due for renewal"
           "No renewals were attempted"
           → Nghĩa là bạn đang chạy nhầm entrypoint renew, cert cho api.hrmgo.site
                CHƯA được tạo. Khi đó hãy chạy lại đúng lệnh có --entrypoint certbot ở trên.

           Nếu thấy warning kiểu:
           "volume ... already exists but was not created by Docker Compose"
           → Có thể tạm bỏ qua ở bước này; đây không phải lỗi chặn việc cấp cert.

6.2. TÌM và SỬA các dòng sau:

     TRƯỚC (cũ):
     ───────────
     ALLOWED_HOSTS=hrmgo.site,www.hrmgo.site
     CORS_ALLOWED_ORIGINS=https://hrmgo.site,https://www.hrmgo.site,https://erp-system-nine-zeta.vercel.app

     SAU (mới):
     ──────────
     ALLOWED_HOSTS=api.hrmgo.site,hrmgo.site,www.hrmgo.site
     CORS_ALLOWED_ORIGINS=https://hrmgo.site,https://www.hrmgo.site

     Giải thích:
     - ALLOWED_HOSTS: thêm api.hrmgo.site (domain mới của backend)
     - CORS_ALLOWED_ORIGINS: chỉ cần domain của frontend (hrmgo.site)
       KHÔNG cần URL vercel cũ (erp-system-nine-zeta.vercel.app) vì
       giờ đã dùng custom domain

     CORS_ALLOWED_ORIGINS=https://hrmgo.site,https://www.hrmgo.site,https://erp-system-nine-zeta.vercel.app

6.6. Test API trực tiếp từ VPS:
     curl -X POST https://api.hrmgo.site/api/auth/login/ \
       -H "Content-Type: application/json" \
       -d '{"username":"admin","password":"admin123"}'

     → Phải trả về JSON chứa access token và refresh token
     → Ví dụ: {"access":"eyJ...","refresh":"eyJ..."}

           CÁCH TEST ĐÚNG:
           1. Chạy đúng lệnh trên ngay trong VPS shell
           2. Không thêm dấu nháy lạ, không xuống dòng sai vị trí
           3. Nếu thành công, output sẽ là JSON một dòng, ví dụ:
                    {"refresh":"eyJ...","access":"eyJ...","user":{...}}
                    hoặc
                    {"access":"eyJ...","refresh":"eyJ..."}
           4. Chỉ cần thấy token `access` hoặc `refresh` là login API đã hoạt động

           CÁCH ĐỌC KẾT QUẢ:
           - Trả về access/refresh token:
                → Thành công hoàn toàn
                → Backend + nginx + SSL + endpoint login đều OK
           - 400 Bad Request:
                → Thường là sai format JSON hoặc sai field request
                → Kiểm tra lại đúng body:
                     {"username":"admin","password":"admin123"}
           - 401 Unauthorized hoặc thông báo sai tài khoản:
                → API chạy bình thường nhưng username/password sai
           - 500 Internal Server Error:
                → Backend có lỗi, xem log backend
           - curl: (7) Couldn't connect to server:
                → nginx/HTTPS chưa chạy đúng, quay lại bước 5.5-5.7

           Nếu muốn xem rõ cả status code lẫn body, dùng thêm lệnh này:
           curl -i -X POST https://api.hrmgo.site/api/auth/login/ \
                -H "Content-Type: application/json" \
                -d '{"username":"admin","password":"admin123"}'

           Nếu muốn debug khi lỗi 500 hoặc 400 không rõ nguyên nhân:
           docker compose logs backend --tail 50

     Nếu lỗi:
           - 400 Bad Request trả về HTML trang Django:
                → Khả năng cao nhất là `api.hrmgo.site` chưa có trong ALLOWED_HOSTS
                → Hoặc bạn đã sửa `.env.production` nhưng chưa rebuild backend
                → Kiểm tra và sửa trên VPS:
                     ALLOWED_HOSTS=api.hrmgo.site,hrmgo.site,www.hrmgo.site
                → Sau đó chạy:
                     docker compose up -d --build backend
                     docker compose restart nginx
                     docker compose logs backend --tail 30
                → Nếu log có dòng kiểu:
                     Invalid HTTP_HOST header: 'api.hrmgo.site'
                     thì đúng là lỗi ALLOWED_HOSTS
           - 400 Bad Request dạng JSON từ API:
                → Thường là sai format request hoặc thiếu field
           - 401 Unauthorized: sai username/password
     - 500: xem logs: docker compose logs backend --tail 30
     - Connection refused: docker compose ps, kiểm tra nginx chạy chưa

  7. Test lại:
       curl -i -X POST https://api.hrmgo.site/api/auth/login/ \
           -H "Content-Type: application/json" \
           -d '{"username":"admin","password":"admin123"}'

Nếu log backend có dòng kiểu:
  Invalid HTTP_HOST header: 'api.hrmgo.site'
thì nguyên nhân đã được xác nhận chính xác là ALLOWED_HOSTS.

  3. docker compose up -d --build backend

─── SỰ CỐ 4: SSL error trên api.hrmgo.site ───

Nguyên nhân: Cert chưa có hoặc hết hạn

─── SỰ CỐ 4D: HTTPS port 443 không kết nối được ───

Nguyên nhân thường gặp nhất: vẫn đang dùng nginx.conf tạm chỉ có HTTP ở bước 5.3
Xử lý:
  1. Mở lại deploy/nginx.conf
  2. Đảm bảo file có block `listen 443 ssl http2;`
  3. Đảm bảo file trỏ tới cert:

─── SỰ CỐ 4C: chạy certbot nhưng chỉ thấy renew hrmgo.site ───

Xử lý:

  2. Dùng lệnh đúng:

           --webroot \
           --webroot-path=/var/www/certbot \
           -d api.hrmgo.site \
           --email your-email@gmail.com \
           --agree-tos \
           --no-eff-email
  3. Sau khi thành công, kiểm tra:
       docker compose exec nginx ls /etc/letsencrypt/live/api.hrmgo.site/

  2. Xóa các dòng kiểu:

Nguyên nhân: wss:// bị chặn hoặc cấu hình sai
Xử lý:
  1. Kiểm tra VITE_WS_URL trong Vercel env: wss://api.hrmgo.site
  2. Test: wscat -c wss://api.hrmgo.site/ws/notifications/
  3. docker compose logs nginx --tail 20 → xem có request /ws/ không

================================================================================
  THỨ TỰ THỰC HIỆN (CHECKLIST)
================================================================================

  □ 1. Cấu hình DNS (thêm A record api → VPS, @ → Vercel)
  □ 2. Đợi DNS propagate (5-30 phút), kiểm tra bằng nslookup
     □ 3. Thêm domain hrmgo.site vào Vercel
     □ 4. Nếu muốn hỗ trợ www thì thêm www.hrmgo.site và cấu hình redirect
     □ 5. Sửa frontend/.env.production trên máy tính
     □ 6. Thêm Vercel Environment Variables
     □ 7. Push code lên GitHub (Vercel tự deploy)
     □ 8. SSH vào VPS: tạo nginx-temp.conf, lấy SSL cert cho api.hrmgo.site
     □ 9. SSH vào VPS: cập nhật deploy/nginx.conf (nội dung bước 4)
     □ 10. SSH vào VPS: sửa backend/.env.production
     □ 11. Rebuild backend + restart nginx
     □ 12. Test CORS từ máy tính (curl)
     □ 13. Test login trên https://hrmgo.site
     □ 14. Cấu hình máy chấm công → api.hrmgo.site:80
